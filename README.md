Mod 8 : Portfolio Reflection 

  Throughout this course, I started to see how interpreting user needs is more than just taking requirements at face value. I think understanding what users actually want requires asking clarifying questions, breaking larger ideas into smaller pieces, and translating those needs into user stories. Writing user stories helped me focus on the user’s perspective instead of just thinking about the technical solution. By structuring features as “As a user, I want…” statements, it made the purpose of each feature clearer and easier to prioritize during sprint planning. 

  When I approach developing programs now, I think more in terms of iteration instead of trying to build everything at once. Agile practices like sprint planning, daily stand-ups, reviews, and retrospectives showed me that progress improves when work is broken into manageable increments. I hope to continue using iterative development, regular feedback, and visible task tracking in my future work. I can see how these practices reduce confusion, improve adaptability, and help teams respond to change without losing direction. 

  To me, being a good team member in software development means communicating clearly, being open to feedback, and taking ownership of assigned work. It also means understanding that writing code is only part of the role. Collaboration, transparency, and accountability are just as important as technical skills. Throughout this project, I saw how structured communication builds trust within a team and helps ensure that everyone is working toward the same goal.
